# Codex 受控执行模式（CVE 项目实践）

> 本文档沉淀的是 Codex 在当前 CVE 项目中的使用方案。
> 它关注的是“如何使用 Codex 作为受控执行引擎”，不是业务目标架构文档，也不是对 Codex 产品本身的完整说明。

## 核心定位

在这套系统里，我们主要把 Codex 当成一个**受控执行引擎**，而不是把系统状态寄托在 Codex 自己身上。

更具体地说，Codex 在这里承担的是：

- 一个可被脚本拉起的 agent runtime
- 一个带 `Web search`、`MCP`、`notify` 的执行容器
- 一个可以通过 `CODEX_HOME`、prompt、env、wrapper 和启动参数被约束住的能力模块

系统真实状态不保存在 Codex 会话上下文中，而是外置在调度系统自己的状态存储里。

## 为什么这样用

如果把系统状态、任务进度、重试逻辑和恢复能力建立在 Codex 自身会话记忆上，会出现几个问题：

- fresh 会话无法自然接管
- worker 异常退出后状态容易丢失
- 行为边界不稳定，难以审计
- 无人值守场景下容易因为交互确认或能力漂移卡住

因此，这里的原则不是“让 Codex 记住系统”，而是“让系统在外部持有真相，再把当前快照喂给 fresh Codex 去执行一轮”。

## 五个核心做法

### 1. 用项目级 `CODEX_HOME` 做环境隔离

我们给这套 workflow 单独准备项目级 `CODEX_HOME`，而不是直接复用默认的 `~/.codex`。

这样隔离出来的是 Codex 运行时边界，包括：

- provider 配置
- MCP 配置
- notify 配置
- Codex 自己的本地状态

这样做的效果是：

- 不污染个人日常使用的 Codex 环境
- 系统只暴露我们允许的配置
- MCP 能力面可以被严格裁剪
- 不同 workflow 可以有独立运行时，不互相串扰

在当前 CVE 项目里，这意味着可以只保留我们需要的 `cveawg` 等能力，而不是开放一个默认的、宽松的 Codex 环境。

### 2. 坚持 fresh `codex exec`，不依赖会话记忆

`mgr` 每一轮都是重新启动的 fresh `codex exec`，`worker` 也是一次性执行单元。

这利用了 Codex 的两个特性：

- `codex exec` 很适合“给定上下文，执行一轮，然后退出”
- fresh 会话天然避免历史记忆漂移

因此我们不是让 Codex 长期记住系统内部状态，而是每次都重新提供：

- 当前任务快照
- 当前运行态
- 固定 prompt
- 固定命令与启动参数

这样做的直接收益是：

- fresh `mgr` 可以随时接管调度
- worker 是短生命周期，故障域更小
- 整个系统恢复时不依赖某个长期存活会话

### 3. 用 prompt 和启动参数收紧能力边界

我们没有让 Codex 在系统里“自由发挥”，而是用固定 prompt 和固定启动参数把能力边界提前收紧。

典型约束包括：

- `worker` 只做单个 CVE 的原始材料定位
- `mgr` 只做巡检、验收、调度，不做深度搜集
- 资料查询必须走 `Web search`
- 看源码需要先 `git clone` 到本地再查看
- 默认使用类似 `approval_policy="never"` 的配置，避免无人值守时卡在权限确认

这里的关键点是：

- Codex 提供的是推理和执行能力
- “能做什么、不能做什么”由系统外部明确规定

也就是说，控制权不在 agent 自主判断里，而在宿主系统的运行时约束里。

这里还有一个更严格的原则：

- worker 拿到的不是“通用工具箱”
- worker 拿到的是“面向该类 task 的隔离工具面”

也就是说，工具不只是被限制，还需要和任务类型相匹配。

### 4. 把真实状态外置到 SQLite，而不是留在 Codex 上下文里

这是整套方案里最关键的设计。

我们没有把“任务做到哪了”寄托在 Codex 上下文，而是把真实状态外置到 SQLite。

外置内容包括：

- 任务状态
- run 生命周期
- heartbeat
- notification
- 重试次数
- 限额冷却状态

这样 Codex 只是状态机的参与者，不是状态的拥有者。

这样做的好处是：

- `mgr` 可以换一个 fresh agent 继续接管
- `worker` 崩了不会丢系统真相
- 系统行为可审计
- 系统状态可恢复、可重放、可收敛

从架构角度看，这一步把“智能执行能力”和“控制面真相”彻底分离开了。

### 5. 使用 Codex 原生机制做异步信号和受控能力扩展

这套方案并没有把 Codex 仅仅当作一个命令执行器，而是实际利用了它的几种原生机制：

- `notify`：worker 完成一个 turn 或结束时向系统发出信号
- `MCP`：只挂载需要的工具面，例如 `cveawg`
- `--search`：把资料搜索统一收敛到 Codex 的 `Web search` 通道
- `codex exec`：把 `mgr` 和 `worker` 做成一次性、可脚本化的执行单元

这些机制组合起来之后，Codex 更像一个“受控 agent runtime”，而不是一个自由聊天界面。

## Task-Scoped Tooling

在这套模式里，worker 的工具边界应该由 task 类型决定，而不是只由运行时默认配置决定。

这意味着同一个 workflow engine 可以面对不同 task 类型装配不同的受控能力面，例如：

- 不同的 prompt 模板
- 不同的 MCP 集合
- 不同的搜索策略
- 不同的源码查看策略
- 不同的环境变量和 wrapper

因此，`CODEX_HOME` 隔离只是第一层。
更完整的约束其实是：

- 任务类型定义需要什么工具
- 运行时适配层只暴露这些工具
- worker 只能在这组 task-scoped capability 里执行

这样做的价值是：

- worker 行为更可预测
- 不同 workflow 之间更容易隔离
- 工具能力和任务语义保持一致
- 宿主系统更容易审计 agent 的执行边界

## 运行时抽象

从系统设计上看，可以把 Codex 在这里的位置抽象为一个运行时适配层：

- 它负责承载 agent 的执行
- 它暴露有限且可控的能力面
- 它通过外部状态存储接收任务上下文
- 它通过通知与状态回写融入宿主系统的控制闭环

所以，Codex 本身不是“系统”，而是系统里的一个执行模块。

## 与 CVE 场景的关系

当前实现落在 CVE 场景中，因此我们看到的是：

- `worker` 处理的是单个 CVE
- 产物是证据包
- 搜索目标是漏洞资料、仓库、补丁、公告等
- MCP 挂的是 CVE 相关能力

但这套使用模式并不天然绑定 CVE 领域。

可迁移的部分包括：

- 项目级 `CODEX_HOME` 隔离
- fresh `codex exec`
- prompt 和启动参数约束
- task-scoped tooling
- 状态外置
- `notify` / `MCP` / `search` 这些原生机制的组合使用

因此，它更适合作为一类通用模式沉淀下来：

**用 Codex 作为受控执行引擎接入宿主调度系统，而不是把宿主系统做进 Codex 会话本身。**

## 一个简化判断标准

如果一个系统满足下面几点，它就比较适合采用这种模式：

- 任务可以被切成短生命周期执行单元
- 系统需要无人值守运行
- 系统需要明确的能力边界
- 系统需要审计、恢复、重试、接管能力
- 系统希望使用 agent 的推理能力，但不希望把控制面交给 agent

## 一句话总结

这套实践的本质不是“让 Codex 成为系统本身”，而是“让 Codex 成为一个被 `CODEX_HOME`、prompt、启动参数、状态外置和原生机制组合共同约束住的受控执行引擎”。
