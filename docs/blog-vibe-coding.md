# Vibe Coding 的质量问题，本质上是一个治理问题

Vibe coding 的质量上限由治理基础设施决定，不由代码审查决定。

我在一个 Clojure 项目上验证了这一点：95 个源文件，90 个测试，89% 的行覆盖率，11 个自动化治理 gate。我没有手动写过一行代码。这不是结论，这是前提——我想讨论的是这件事为什么能发生，以及它在架构上意味着什么。

---

## 审查为什么不够用

大多数人对 AI 生成代码的直觉反应是：加强审查。仔细看每一段输出，发现问题就打回去重来。

这个直觉在低频率下是对的。但当 AI 成为主要的代码生产者时，审查的速度跟不上生成的速度。更根本的问题是：审查是在检查结果，而结果是无穷的——每次生成都不同，每次都要重新判断。

真正有效的质量保障不在结果上，在约束上。约束是有限的，结果是无限的。把约束写清楚、写成可执行的规则，让规则去检查每一次生成的结果，才是可扩展的路径。

这就是为什么 vibe coding 的核心问题是治理问题，不是代码质量问题。你需要的不是更好的眼力，而是更清晰的约束系统。

---

## 治理不能是事后的

这是大多数项目做错的地方。通常的节奏是：先写业务代码，等技术债积累到一定程度，再专门投入时间加 linting、加规则、加覆盖率要求。这个模式在传统开发里已经很难成功，在 AI 辅助开发里几乎必然失败——AI 的生产速度意味着债务积累的速度也是 10 倍的。

有效的治理必须从第一天开始，和业务逻辑同步演进。发现一个新的结构性问题，就在同一轮迭代里把它编码成规则。规则不是约束清单，是判断的文档化。

这个项目的 `bb check` 并行运行 11 个 gate，覆盖格式、静态分析、文件结构、前端架构、CSS 规范、编译、测试、覆盖率。每条规则都不是批量加进去的，而是在遇到具体问题时一条一条长出来的。

关键的设计决策是双阈值：每条结构性规则都有 warn 和 fail 两条线，比如文件长度 240 行警告、300 行阻断。这不是保险措施，而是故意留出的压力窗口。超过 warn 时你知道问题在积累，但还没有被迫停下来——这个区间的作用是让结构性问题可见，而不是让它们隐形直到爆发。

```
WARNING [file-length-governance] src/meta_flow/cli/defs.clj has 286 lines,
exceeding the warning threshold of 240 lines.
Intent: use file length as a governance signal to avoid a single namespace
accumulating too much responsibility.
```

注意 `Intent:` 这个字段。每条规则都要求写清楚它存在的理由，而不只是阈值数字。这对 AI 特别重要：当规则解释了为什么，AI 在生成代码时会把这个约束内化为设计目标，而不是单纯需要通过的检查。不懂意图的规则会被绕过，懂意图的规则会被遵守。

## 一条 lint 规则同时是规格、检测器和任务单

这是整套工作流里最反直觉的部分，也是摩擦最低的部分。

通常的假设是：先写代码，再写规则验证代码是否正确。但这个顺序在 AI 辅助开发里可以完全倒过来——而且倒过来之后，整个流程变了。

具体的操作是这样的：当你意识到代码库里某种结构开始往错误方向漂移，或者预判到 AI 在某类问题上会犯一致性的错误，你先写一条 lint 规则来检测这个问题。规则写完，立刻跑一次——它会找出代码库里所有现存的违规点。然后你把这个输出告诉 AI：修复这些。

不需要制定计划，不需要逐文件 review，不需要向 AI 解释问题在哪里。lint 规则已经完成了定位工作，AI 只需要执行修复，再跑一次确认规则通过。

这个循环之所以成立，是因为 lint 规则同时承担了三个角色：它是对"正确代码"的规格说明，是在整个代码库里自动检测违规的工具，也是给 AI 的任务单。三件事是同一件事。

`bb check` 的输出格式是机器可读的，正是为了让这个循环尽可能短：

```
AI governance summary: warning
warning gates:
- structure-governance: 6 warning(s), 0 error(s), 5 file-length issue(s)
  evidence: src/meta_flow/cli/defs.clj has 286 lines (threshold 240)
  action: Split oversized namespaces or directories by responsibility.
passed gates:
- static-analysis: no static-analysis findings
- executable-correctness: 312 tests passed
- coverage-governance: overall line coverage 89.06%
```

每个 gate 有 `status`、`evidence`、`action`，可以直接粘贴进对话。AI 拿到这个输出，知道问题在哪、影响是什么、该怎么处理。

一旦某类问题被编码进规则，你就不需要再为这类问题做人工 review 了。规则在跑，问题会被自动发现。你的注意力可以移向下一个还没被规则覆盖的问题。这是治理系统真正的复利：每条规则都在帮你把精力从已知问题上释放出来。

这类规则最有价值的地方，是它可以强制执行**拓扑约束**——关于代码应该在哪里、依赖应该流向哪里，而不只是代码写得对不对。

举一个具体的例子：这个项目的共享 UI 组件治理。规则要求所有共享 UI 实现只能存在于三个目录：`ui/layout`、`ui/patterns`、`ui/primitives`。AI 在生成共享组件时，会倾向于把它放在最近工作的上下文里——某个功能目录，或者直接放进公共入口文件。这在局部是合理的，但全局上会让"共享组件"散落在整个代码库里。

治理系统用四条联锁规则把这个问题关死：

**位置 + 命名空间双重验证。** 每个落在 `shared-ui-root` 下的文件，不只检查它在哪个目录，还读取文件内的 `ns` 声明，验证命名空间前缀和目录是否一致。文件在对的地方但 namespace 声明错了，也会报错。两个维度同时校验，没有逃脱路径。

**facade 纯度检查。** `components.cljs` 是共享组件的对外入口。治理规则逐个读取这个文件里的每一个 top-level form，只允许两种存在：`ns` 声明，以及 `(def Foo some-ns/Foo)` 这种纯符号别名。有任何实现逻辑——哪怕是一个本地 helper 函数——就立刻报错。这保证了入口文件永远是一张索引，所有实现都在正确的子目录里。

**依赖方向检查。** 规则读取每个共享文件的 `require` 列表，验证没有向上依赖页面命名空间或状态命名空间。共享层对具体页面产生依赖，是最常见的架构腐化路径之一，AI 在"顺手"引入某个已有功能时特别容易触发这类问题。

这四条规则共同锁住同一个不变式，从文件系统位置、命名空间声明、公共 API 结构、依赖关系四个维度同时验证。AI 在这个约束下生成代码，没有办法"顺手"破坏共享层的结构——不是因为它在努力遵守规则，而是因为任何偏差都会被立刻检测到，然后作为任务单反馈回来修复。

## 语言选择在这里是结构性问题

说到这里必须谈语言选择，因为这不是偏好问题，是摩擦问题。

TypeScript 能写 ESLint 自定义规则，Python 有 `ast` 模块，Go 有 `golang.org/x/tools/go/analysis`。技术上这些语言都能做程序分析。但它们有一个共同的结构性问题：**治理工具和业务代码是两套语言、两套心智模型、两套工具链。**

写一条 ESLint 规则，你需要理解 ESLint 的 visitor 模式、AST 节点类型、selector 语法。这些跟你写 React 组件时用的思维完全不同。结果是：写治理规则的成本远高于写业务代码，这种成本差异决定了治理规则在实践中会被推迟——推迟到它变成一次性的大型投资，而不是持续的小投入。

Clojure 里这个成本差异不存在。

添加一条新治理规则，就是写一个返回普通 map 的函数，和写业务逻辑没有任何形式上的区别：

```clojure
(defn my-gate []
  {:gate    :my-rule
   :status  (if (check-something) :pass :error)
   :headline "..."
   :action  "..."})
```

这不是语法糖，背后有一个更深的原因。

Lisp 有一个在其他语言里不存在的性质：**同像性**——代码和数据使用同一种表示。一段 Clojure 程序在被执行之前，它首先是一个普通的 Clojure 数据结构：列表、符号、向量。这意味着读取一个 Clojure 文件，和读取一段业务数据，是完全相同的操作。

这个项目里有一条治理规则，用来检查 `components.cljs` 是否只是一个薄的 re-export facade，没有混入实现逻辑。核心的解析代码是这样的：

```clojure
(defn- read-top-level-forms [path]
  (with-open [reader (java.io.PushbackReader. (io/reader path))]
    (binding [*read-eval* false]
      (loop [forms []]
        (let [form (read {:eof ::eof} reader)]
          (if (= ::eof form)
            forms
            (recur (conj forms form))))))))
```

`read` 是 Clojure 内置的函数，它把源文件解析成原生数据结构——就是普通的列表和符号。读完之后，检查每个 top-level form 是否符合 facade 的结构要求：

```clojure
(defn- allowed-facade-form? [shared-ui-root form]
  (or (and (seq? form) (= 'ns (first form)))          ;; ns 声明 — 允许
      (and (seq? form)
           (= 'def (first form))                       ;; def 形式
           (= 3 (count form))                          ;; 只有三个元素：def、名字、值
           (symbol? (second form))                     ;; 名字是符号
           (symbol? (nth form 2))                      ;; 值也是符号（说明是引用，不是实现）
           (facade-target-shared-ui? shared-ui-root    ;; 引用的是 shared-ui 层
                                     (nth form 2)))))
```

这段代码在操作的是程序的语法树，但它看起来和操作任何其他数据结构完全一样。没有引入外部解析库，没有学习新的 API，就是模式匹配和数据查询。

在 TypeScript 里做同样的事情，你需要 TypeScript Compiler API 或者 tree-sitter。这两个工具都有自己的 AST 节点类型、自己的 visitor 模式、自己的查询语法——和你写 TypeScript 业务代码的方式完全不同。你在操作"关于代码的数据"，而不是代码本身。

这个阻抗差异决定了什么样的治理规则会被真正写出来。高摩擦意味着只有最重要的规则才会有人投入时间去写，其他的就靠约定和文档——而约定和文档在 AI 大量生成代码的情况下是靠不住的。

**摩擦系数决定哪些良好实践会真正发生。**

## 这不是没有代价的

说了这么多优势，也要说清楚代价。

AI 对 Clojure 的生成质量显著低于 Python 和 TypeScript，因为训练数据规模的差距是真实的。这意味着更多的修正轮次，更高的迭代成本。如果你选 Clojure，这个代价你必须接受。

更根本的问题是：这套方法要求你在开始之前就对问题域有足够清晰的理解，知道什么约束值得编码。如果你对业务逻辑本身都还没有想清楚，治理规则会写错方向，或者根本写不出来。AI 生成代码的速度换不回架构判断力。

目前这个项目有 5 个文件在 file-length warning 区，`defs/` 目录的宽度刚过阈值，覆盖率离 warn 线只有 1 个百分点。这些压力点集中在 `defs/` 这条线——一个正在扩张的子域，结构还没有来得及跟上。这不是治理失效，这是治理在工作：它让结构债可见、可定位、有量化读数。下一步该在哪里做结构分解，很清楚。

混乱的代码库里，问题是弥散的，你不知道该从哪里开始。治理系统的价值不是消除问题，而是让问题有地址。

---

最后说一件也许不那么显然的事。

Vibe coding 这个词让人觉得你在放弃什么——放弃控制，放弃理解，换取速度。我的经验是：你放弃的只是实现，你保留的是判断。这两件事以前是捆绑在一起的，因为没有人能帮你实现。现在它们可以分开。

判断是有限的，实现是无限的。AI 接管了无限的那一侧，你需要做好有限的那一侧。协议边界在哪里划，哪个方向的技术债现在就要阻断、哪个可以先 warn，治理规则背后的意图是什么——这些是判断，不是代码。

质量来自判断，不来自手写。
